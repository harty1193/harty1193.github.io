Note 01

Automating BitLocker Recovery Key Backup to the Cloud

June 2026  ·  Hrudananda Das  ·  8 min read
BitLocker PowerShell Entra ID Active Directory MSA MBAM

BitLocker is enabled on millions of enterprise Windows devices — but enabling it is only half the story. The recovery password sitting in a key protector is useless if it lives nowhere retrievable. I've seen this failure mode on the escalation floor more times than I can count: device gets bricked at the pre-boot screen, and nobody can find the recovery key.

This note covers how to automate recovery key backup across all three destinations — Entra ID (Azure AD), on-premises Active Directory, and Microsoft Account (MSA) — using the right PowerShell cmdlet for each scenario, plus a production-ready script that handles the common messy reality of multiple key protectors on one drive.

Prerequisites BitLocker must already be enabled on the target drive. The scripts below back up an existing recovery password — they don't enable BitLocker from scratch. Run PowerShell as Administrator.

The Three Backup Cmdlets

Microsoft provides three distinct cmdlets for backing up a BitLocker recovery password, each targeting a different destination. Choosing the wrong one for your environment is the most common mistake.

Cmdlet Destination Scenario Requires
BackupToAAD-BitLockerKeyProtector Entra ID / AAD Cloud-joined or hybrid devices managed by Intune / Entra Device registered in Entra ID; internet access
Backup-BitLockerKeyProtector On-prem AD Domain-joined devices with MBAM or Group Policy backup AD write access; schema extended for BitLocker
BackupToMSA-BitLockerKeyProtector Microsoft Account Consumer / personal devices signed in with an MSA Active MSA signed in to Windows; internet access
Naming quirk Notice the inconsistency: the AD cmdlet is Backup-BitLockerKeyProtector (with a hyphen and noun), while the AAD and MSA variants use a verb-prefix style (BackupToAAD-, BackupToMSA-). This reflects their different introduction eras and is a common source of confusion.

Backup to Entra ID (AAD)

This is the most common enterprise scenario for modern, cloud-managed deployments. The key is stored in the device object inside Entra ID and is visible to admins in the Microsoft Entra portal or via Microsoft Intune.

Syntax

PowerShell 
# Get the BitLocker volume
$vol = Get-BitLockerVolume -MountPoint "C:"

# Get the RecoveryPassword protector ID
$kp = $vol.KeyProtector | Where-Object { $_.KeyProtectorType -eq 'RecoveryPassword' }

# Back up to Entra ID (AAD)
BackupToAAD-BitLockerKeyProtector -MountPoint "C:" -KeyProtectorId $kp.KeyProtectorId
Verify in the portal After running: Entra Admin Center → Devices → All Devices → select device → BitLocker Keys. The key should appear within a few minutes.

Backup to On-Premises Active Directory

For domain-joined devices without cloud management, the recovery password can be written directly into the computer object in AD DS. This requires the AD schema to have the BitLocker attribute extensions — available from Windows Server 2008 R2 onwards, or via MBAM.

Syntax

PowerShell 
$vol = Get-BitLockerVolume -MountPoint "C:"
$kp  = $vol.KeyProtector | Where-Object { $_.KeyProtectorType -eq 'RecoveryPassword' }

# Back up to on-premises Active Directory
Backup-BitLockerKeyProtector -MountPoint "C:" -KeyProtectorId $kp.KeyProtectorId
Verify in AD Open ADUC → find the computer object → Properties → BitLocker Recovery tab. If the tab is missing, install the BitLocker Recovery Password Viewer RSAT feature on your admin workstation.

Backup to Microsoft Account (MSA)

For consumer or personal Windows devices where the user is signed in with a Microsoft Account, the recovery key can be backed up to their account at account.microsoft.com/devices/recoverykey.

Syntax

PowerShell 
$vol = Get-BitLockerVolume -MountPoint "C:"
$kp  = $vol.KeyProtector | Where-Object { $_.KeyProtectorType -eq 'RecoveryPassword' }

# Back up to Microsoft Account
BackupToMSA-BitLockerKeyProtector -MountPoint "C:" -KeyProtectorId $kp.KeyProtectorId

Production Script — AAD Backup with Multi-Key Handling

Real-world machines often accumulate multiple RecoveryPassword protectors over time — after OS upgrades, re-imaging, or manual key additions. Blindly backing up all of them adds noise to Entra ID. The script below handles this cleanly: it lets you select which key to back up, backs that one up to Entra ID, and optionally removes the rest.

What the script does — step by step

1
Input validation — Accepts a drive letter in either C or C: format, normalises it, and verifies the drive exists before doing anything.
2
Cmdlet availability check — Confirms that Get-BitLockerVolume, BackupToAAD-BitLockerKeyProtector, and Remove-BitLockerKeyProtector are all present on the machine. Exits cleanly if any are missing.
3
Key protector discovery — Retrieves the BitLocker volume and filters for RecoveryPassword type protectors only.
4
Single vs. multiple key handling — If only one recovery key exists, uses it automatically. If multiple exist, lists them and prompts the user to choose one to keep.
5
Backup to Entra ID — Calls BackupToAAD-BitLockerKeyProtector with the selected key protector ID. Fails with a clear error if the backup fails.
6
Cleanup of stale keys — Lists remaining (unselected) recovery protectors and asks for explicit YES confirmation before removing each one.
7
Summary output — Reports which key was kept and backed up, which were removed, and which failed to remove — colour-coded for at-a-glance reading.
Warning — key removal is irreversible Removing a RecoveryPassword protector with Remove-BitLockerKeyProtector is permanent. Always confirm the backup completed successfully in Entra ID before allowing removal. The script does not remove keys without explicit YES confirmation.

Full script

PowerShell — Backup-BitLockerToAAD.ps1 
try {
    # Prompt user input
    $driveLetter = Read-Host "Enter the drive letter (e.g., C or C:)"

    # Normalize input (convert C -> C:)
    if ($driveLetter -match '^[A-Za-z]$') {
        $driveLetter = "${driveLetter}:"
    }

    # Validate format
    if ($driveLetter -notmatch '^[A-Za-z]:$') {
        throw "Invalid drive format. Please enter like C or C:."
    }

    # Check if drive exists
    if (-not (Test-Path $driveLetter)) {
        throw "Drive $driveLetter does not exist."
    }

    # Check required cmdlets
    if (-not (Get-Command Get-BitLockerVolume -ErrorAction SilentlyContinue)) {
        throw "Get-BitLockerVolume cmdlet is not available on this system."
    }
    if (-not (Get-Command BackupToAAD-BitLockerKeyProtector -ErrorAction SilentlyContinue)) {
        throw "BackupToAAD-BitLockerKeyProtector cmdlet is not available on this system."
    }
    if (-not (Get-Command Remove-BitLockerKeyProtector -ErrorAction SilentlyContinue)) {
        throw "Remove-BitLockerKeyProtector cmdlet is not available on this system."
    }

    # Get BitLocker volume
    $volume = Get-BitLockerVolume -MountPoint $driveLetter -ErrorAction Stop

    if ($null -eq $volume) {
        throw "No BitLocker volume information found for $driveLetter."
    }

    # Filter only RecoveryPassword protectors
    $recoveryKPs = @($volume.KeyProtector | Where-Object {
        $_.KeyProtectorType -eq 'RecoveryPassword'
    })

    if (-not $recoveryKPs -or $recoveryKPs.Count -eq 0) {
        throw "No RecoveryPassword protector found on $driveLetter."
    }

    $selectedKP = $null

    if ($recoveryKPs.Count -eq 1) {
        $selectedKP = $recoveryKPs[0]
        Write-Host "Only one RecoveryPassword protector found. It will be used." -ForegroundColor Cyan
        Write-Host "Selected KeyProtectorId: $($selectedKP.KeyProtectorId)" -ForegroundColor Cyan
    }
    else {
        Write-Host ""
        Write-Host "Multiple RecoveryPassword protectors found on $driveLetter" -ForegroundColor Yellow

        for ($i = 0; $i -lt $recoveryKPs.Count; $i++) {
            Write-Host ("[{0}] {1}" -f ($i + 1), $recoveryKPs[$i].KeyProtectorId)
        }

        do {
            $selection = Read-Host "Enter the number of the RecoveryPassword protector to back up and keep"
            $validSelection = $selection -match '^\d+$' -and
                              [int]$selection -ge 1 -and
                              [int]$selection -le $recoveryKPs.Count

            if (-not $validSelection) {
                Write-Host "Invalid selection. Please enter a number between 1 and $($recoveryKPs.Count)." -ForegroundColor Red
            }
        } until ($validSelection)

        $selectedKP = $recoveryKPs[[int]$selection - 1]
        Write-Host ""
        Write-Host "Selected KeyProtectorId: $($selectedKP.KeyProtectorId)" -ForegroundColor Cyan
    }

    # Backup selected protector to AAD
    Write-Host ""
    Write-Host "Backing up selected RecoveryPassword protector to AAD..." -ForegroundColor Cyan

    BackupToAAD-BitLockerKeyProtector `
        -MountPoint $driveLetter `
        -KeyProtectorId $selectedKP.KeyProtectorId `
        -ErrorAction Stop

    Write-Host "Backup to AAD completed successfully for KeyProtectorId: $($selectedKP.KeyProtectorId)" -ForegroundColor Green

    # Identify other recovery protectors
    $otherRecoveryKPs = @($recoveryKPs | Where-Object {
        $_.KeyProtectorId -ne $selectedKP.KeyProtectorId
    })

    if ($otherRecoveryKPs.Count -gt 0) {
        Write-Host ""
        Write-Host "The following RecoveryPassword protector(s) are not selected and can be removed:" -ForegroundColor Yellow
        foreach ($kp in $otherRecoveryKPs) {
            Write-Host " - $($kp.KeyProtectorId)"
        }

        $confirmRemoval = Read-Host "Type YES to remove the above RecoveryPassword protector(s), or anything else to keep them"

        if ($confirmRemoval -eq 'YES') {
            $removedIds = @()
            $failedIds  = @()

            foreach ($kp in $otherRecoveryKPs) {
                try {
                    Remove-BitLockerKeyProtector `
                        -MountPoint $driveLetter `
                        -KeyProtectorId $kp.KeyProtectorId `
                        -ErrorAction Stop

                    $removedIds += $kp.KeyProtectorId
                    Write-Host "Removed KeyProtectorId: $($kp.KeyProtectorId)" -ForegroundColor Green
                }
                catch {
                    $failedIds += $kp.KeyProtectorId
                    Write-Host "Failed to remove KeyProtectorId: $($kp.KeyProtectorId). Error: $($_.Exception.Message)" -ForegroundColor Red
                }
            }

            Write-Host ""
            Write-Host "Removal summary:" -ForegroundColor Cyan
            Write-Host " Kept and backed up : $($selectedKP.KeyProtectorId)"

            if ($removedIds.Count -gt 0) {
                Write-Host " Removed            : $($removedIds -join ', ')" -ForegroundColor Green
            }
            if ($failedIds.Count -gt 0) {
                Write-Host " Failed to remove   : $($failedIds -join ', ')" -ForegroundColor Red
            }
        }
        else {
            Write-Host "Removal skipped. Other RecoveryPassword protector(s) were not removed." -ForegroundColor Yellow
        }
    }
    else {
        Write-Host "No extra RecoveryPassword protectors found to remove." -ForegroundColor Green
    }
}
catch {
    Write-Host ""
    Write-Host "Error: $($_.Exception.Message)" -ForegroundColor Red
}

When to Use Which Backup Destination

  • Entra ID — Default choice for any modern managed device enrolled in Intune or joined to Entra. Keys are visible to admins without needing RSAT or AD access.
  • On-prem AD — Use when devices are domain-joined and you have MBAM or GPO-enforced key backup. Necessary for air-gapped or hybrid environments without cloud management.
  • MSA — Appropriate only for consumer Windows Home/Pro devices where a Microsoft Account is the primary identity. Not suitable for enterprise deployments.
Hybrid environments For hybrid AD-joined devices (joined to both on-prem AD and Entra ID), you can run both Backup-BitLockerKeyProtector and BackupToAAD-BitLockerKeyProtector to ensure the key lands in both places. Back up to AAD first, verify, then back up to AD.

Common Issues & Fixes

ErrorLikely CauseFix
BackupToAAD-... is not recognised Windows version too old or cmdlet not loaded Requires Windows 10 1703+ with BitLocker module. Run Import-Module BitLocker first.
Backup succeeds but key not visible in Entra portal Device not registered / replication delay Confirm device is Entra-joined with dsregcmd /status. Wait up to 15 minutes for replication.
Access denied on AD backup Computer object lacks write permissions to msTPM-OwnerInformation attribute Check AD delegation for BitLocker key backup; verify via MBAM or GPO Store BitLocker recovery information in AD DS.
No RecoveryPassword protector found BitLocker enabled with TPM-only, no recovery password added Run Add-BitLockerKeyProtector -MountPoint C: -RecoveryPasswordProtector to generate one first.
